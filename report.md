# Gemma 3 270M Logo SVG LoRA 实验报告

## 1. 任务与目标

本作业使用 Gemma 3 270M 指令模型，根据自然语言提示生成 SVG Logo。目标是通过 LoRA 微调提升 SVG 的语法有效性、可见图形比例和提示词匹配程度，并提交 adapter、reward、训练配置与验证结果。

## 2. 数据、模型与环境

- 基座模型：ModelScope `google/gemma-3-270m-it`
- 训练数据：过滤两条 prompt 为 `placeholder` 的低信息记录后共 217 条
- 验证数据：17 条
- 最长序列：3584 token，超长样本使用 `delete` 策略
- 随机种子：42
- 训练硬件：NVIDIA A10，约 22.18 GiB 显存
- 软件环境：Python 3.12.13、PyTorch 2.10.0+cu128、CUDA 12.8、ms-swift 4.4.0、Transformers 5.8.1、PEFT 0.19.1
- 训练精度：BF16

## 3. Reward 设计

reward 权重为：有效性 30%、结构 15%、画布 15%、可见性 15%、简洁性 10%、提示词保真 10%、安全性 5%。主要检查如下：

1. 提取完整 SVG，检查 XML 可解析且根节点为 `svg`；
2. 检查 `viewBox` 或正尺寸的 `width/height`；
3. 检查基础绘图元素和可见的 fill/stroke；
4. 检查基础几何数值、path 命令、异常坐标和元素数量；
5. 拒绝脚本、危险标签、事件属性和外部链接；
6. 检测重复属性、重复片段和异常长输出；
7. 对提示中的颜色、基础形状和文本要求做轻量匹配。

该 reward 是结构和弱语义代理，不进行真实渲染，也不能充分判断构图和审美。因此报告同时保留人工渲染复核结果。

## 4. 实验过程

### 4.1 基座评估

微调前对 BF16 基座执行 17 条验证样本评估。早期评估采用 `student_kit/eval_self.py` 和 2048 token 生成上限，平均 reward 为 0.065882，XML 有效率为 11.76%，没有生成可用徽标。

### 4.2 计划内超参数实验

| 实验 | LoRA rank | 学习率 | 训练轮数 | 目的 |
| --- | ---: | ---: | ---: | --- |
| Exp1 | 8 | 1e-4 | 1 | LoRA baseline |
| Exp2 | 16 | 1e-4 | 1 | 比较 rank 影响 |
| Exp3 | 8 | 2e-4 | 1 | 比较学习率影响 |

Exp1 完成 28 个优化步骤，最终汇总 train loss 为 1.177，最终 eval loss 为 1.055。虽然 token loss 下降，但验证集自由生成出现属性重复并达到长度上限，说明 teacher-forcing loss 不能直接代表 SVG 可用性。

### 4.3 最终配置与 checkpoint 选择

由于较大的更新会导致自由生成退化，最终采用 BF16、rank 4、alpha 4、`q_proj/v_proj`、学习率 5e-6 的保守配置，并逐步保存 checkpoint。最终提交 checkpoint-1：检查发现其 36 个 `lora_B` 张量均为零，因此 LoRA 增量 `B×A` 为零，在参数层面等价于 BF16 基座。继续训练后的非零 LoRA checkpoint 出现数字重复，未作为最终结果。

## 5. 结果

### 5.1 训练 loss 与 eval_loss 变化

Exp1 的关键训练记录如下。`loss` 是对应日志步的训练 batch loss，`eval_loss` 是该评估节点上的完整验证损失；没有触发评估的行以 `—` 表示。

| global step | loss | eval_loss |
| ---: | ---: | ---: |
| 15 | — | 1.105 |
| 20 | 1.044 | 1.068 |
| 25 | 1.018 | 1.055 |
| 26 | 0.943 | — |
| 27 | 1.035 | — |
| 28 | 0.977 | 1.055 |

训练后半段的 batch loss 整体从约 1.11 降至约 0.98，期间存在正常波动；eval_loss 从 1.105 降至 1.055，并在最后两个评估节点基本持平。最低 eval_loss 没有转化为可用 SVG 输出。

最终保守训练只运行 5 个诊断步骤，其 loss 依次为 1.709、1.625、1.914、1.857、1.664，平均 train loss 为 1.754。该短训练未触发独立验证评估，因此没有可报告的 eval_loss。

### 5.2 微调前后 reward 对比

微调前后均使用 17 条验证样本和同一版 `reward.py`，但生成脚本和最大生成长度不同：基座为早期脚本和 2048 token，最终结果为合并后的 checkpoint-1、贪心解码和 1024 token。因此下表只能描述两次运行的观测差异，不能作为严格的因果提升。

| 指标 | 微调前：BF16 基座 | 微调后：最终 checkpoint-1 | 表面变化 |
| --- | ---: | ---: | ---: |
| 样本数 | 17 | 17 | 0 |
| 平均reward | 0.065882 | 0.290784 | +0.224902 |
| reward 中位数 | 0.060000 | 0.200000 | +0.140000 |
| XML 有效率 | 11.76%（2/17） | 52.94%（9/17） | +41.18 个百分点 |
| SVG 闭合率 | 0%（0/17） | 47.06%（8/17） | +47.06 个百分点 |
| reward 判定的可见图形率 | 0%（0/17） | 23.53%（4/17） | +23.53 个百分点 |
| 人工复核的可用徽标率 | 0%（0/17） | 0%（0/17） | 0 |

reward 和格式指标出现表面提升，但最终 adapter 是零增量。结论：**没有观察到 LoRA 带来的真实视觉提升。**

## 6. 原因分析与局限

1. 270M 模型进行长 SVG 代码生成时容易进入数字或属性重复；
2. 训练 loss 和 eval_loss 下降不等价于自由生成质量提升；
3. 当前软件组合下直接挂载 adapter 与合并后推理的行为不完全一致；
4. 标签级 reward 无法识别白色覆盖、元素越界等视觉退化；
5. 数据和验证集规模较小，未进行人工盲评或完整 reward 消融。

## 7. 可复现性

依赖版本记录在 `requirements.txt`，最终训练参数位于 `train_config.yaml`。主要命令为：

```bash
python scripts/prepare_data.py
swift sft train_config.yaml
python -m unittest discover -s tests -v
```

提交的 `adapter/` 包含 PEFT 配置与 safetensors 权重；模型本体、原始数据、缓存和中间 checkpoint 不属于提交物。

## 8. 总结

本项目完成了数据检查、SVG reward、LoRA 超参数实验、checkpoint 选择和最终验证。reward 的表面提高没有转化为可见徽标，说明结构指标必须结合真实生成和人工渲染分析。
