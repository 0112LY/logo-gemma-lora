# Gemma 3 270M SVG 徽标 LoRA 微调报告

## 1. 任务与实验设置

本项目使用 Gemma 3 270M 指令模型，根据自然语言提示生成 SVG 徽标。目标不是达到大型模型的视觉质量，而是设计可解释的程序化 reward，并检验 LoRA 是否相对同一基座模型改善 SVG 有效性、可见性和提示词保真度。

- 基座模型：ModelScope `google/gemma-3-270m-it`
- 训练集：过滤 2 条包含 `placeholder` 的记录后共 217 条
- 验证集：17 条
- 框架：ms-swift 4.4.0，PEFT 0.19.1
- 环境：Python 3.12.13，PyTorch 2.10.0+cu128，CUDA 12.8
- GPU：NVIDIA A10，约 22.18 GiB 显存
- 精度：BF16
- 随机种子：42

最终训练使用 rank 4、alpha 4，只在 `q_proj` 和 `v_proj` 上添加 LoRA；学习率为 `1e-6`，常数调度，梯度累积 8，最大长度 3584。提交的 `train_config.yaml` 是训练的准确配置。

## 2. Reward 设计

Reward 由七项组成：有效性 30%、结构 15%、画布 15%、可见性 15%、简洁性 10%、提示词保真度 10%、安全性 5%。实现位于 `student_kit/reward.py`，根目录 `reward.py` 提供入口。

主要检查包括：

1. 提取完整 `<svg>...</svg>`，检查 XML 可解析且根节点为 `svg`；
2. 检查 `viewBox` 或正尺寸的 `width/height`；
3. 检查基础绘图元素及可见的 `fill`、`stroke` 或样式；
4. 检查几何数值、基础 path 命令、异常坐标和元素数量；
5. 拒绝脚本、危险标签、事件属性及外部链接；
6. 检测重复属性、重复片段和异常长输出；
7. 对提示中的颜色、基础形状和文本要求进行轻量匹配。

该指标是训练代理而非真实视觉评分。它能可靠发现未闭合、不可解析、空白或重复退化的 SVG，但不能充分评价构图、美感和复杂语义。

## 3. 实验过程

最初测试了 rank 8/16 和学习率 `1e-4`/`2e-4`。这些较大更新虽然降低了 teacher-forcing loss，却使自由生成进入属性或数字重复，常常达到长度上限而没有闭合 SVG。因此追加保守实验：rank 4、只训练 `q_proj/v_proj`，并将学习率降低到 `1e-6`。

保守实验逐步保存并评估 checkpoint。候选搜索最终延长到 5 个 epoch，并对后期低 `eval_loss` 候选进行完整 17 条验证。最终没有直接采用最低 `eval_loss` 的 checkpoint，而是按完整验证集 reward、XML 有效率、可见绘图率依次选择。

## 4. Loss 与 checkpoint 选择

实验前 20 步的代表性记录如下。单批训练 loss 波动明显，而 `eval_loss` 仅缓慢变化。

| Step | Train loss | Eval loss |
| ---: | ---: | ---: |
| 1 | 1.708534 | 1.761972 |
| 3 | 1.916087 | 1.760738 |
| 5 | 1.665825 | 1.760219 |
| 9 | 1.832760 | 1.759860 |
| 12 | 1.670695 | 1.759037 |
| 14 | 1.642602 | 1.759278 |
| 18 | 1.693991 | 1.758762 |
| 20 | 1.772780 | 1.760627 |

继续训练后，checkpoint-55 的 `eval_loss` 为 1.756222。五个 epoch 内最低的几个候选及其完整生成评测如下：

| Checkpoint | Eval loss | 平均 reward | XML 有效率 | 可见绘图率 |
| ---: | ---: | ---: | ---: | ---: |
| 55 | 1.756222 | **0.141176** | **0.294118** | **0.058824** |
| 125 | 1.754632 | 0.094118 | 0.294118 | 0.000000 |
| 129 | **1.754379** | 0.078824 | 0.235294 | 0.000000 |
| 135 | 1.754669 | 0.042353 | 0.000000 | 0.000000 |

checkpoint-129 的 `eval_loss` 最低，但 SVG reward 明显低于 checkpoint-55；checkpoint-135 甚至完全失去有效 XML。这说明 token 级验证损失与自回归生成的结构稳定性并不一致。最终选择 checkpoint-55，而不是训练末尾或最低 `eval_loss` checkpoint。

## 5. 基座与微调结果

基座和微调模型均使用同一份 17 条验证集、同一 reward、BF16、seed 42、贪心解码、`num_beams=1`、`max_new_tokens=1024`、`repetition_penalty=1.0`，并在生成 `</svg>` 后停止。完整汇总保存在 `results.json`。

| 指标 | BF16 基座 | checkpoint-55 | 变化 |
| --- | ---: | ---: | ---: |
| 样本数 | 17 | 17 | 0 |
| 平均 reward | 0.065882 | **0.141176** | **+0.075294** |
| reward 中位数 | 0.060000 | 0.060000 | 0 |
| XML 有效率 | 0.117647（2/17） | **0.294118（5/17）** | **+0.176471** |
| 可见绘图率 | 0（0/17） | **0.058824（1/17）** | **+0.058824** |

平均 reward 相对基座约提高 114.29%。有效 XML 从 2 个增加到 5 个，并首次出现 reward 判定为可见绘图的输出，因此观察到了相对基座的程序化指标提升。

另一方面，两组 reward 中位数均为 0.06，17 条中只有少数输出贡献了主要增量。这说明提升尚不稳定，不能据此声称真实视觉质量显著提高。隐藏视觉评审可能不认可简单元素堆叠或弱语义匹配。

## 6. 结论与局限

LoRA 的保守更新改善了部分样本的 SVG 闭合和 XML 有效性，checkpoint-55 在固定验证流程中优于基座。但小模型仍经常产生空图、非法属性、重复序列或与提示不一致的图形。继续降低 `eval_loss` 没有持续改善自由生成，反而使后期 checkpoint 的 reward 和可见率下降。

本实验的主要结论是：对代码式生成任务，checkpoint 选择不能只依赖 teacher-forcing loss；必须用固定解码对完整验证集进行结构和可见性评测。同时，程序化 reward 的提升只代表代理目标改善，不能替代人工视觉评价。

## 7. 复现

```bash
pip install -r requirements.txt
python scripts/prepare_data.py
swift sft train_config.yaml
python -m unittest discover -s tests -v
```

训练完成后，使用 `student_kit/eval_self.py` 在相同解码设置下分别评测基座和 adapter，即可复现 `results.json` 中的核心对比。
